# Domain Factory Template (Hugo)

This repository is a **reusable Domain Factory Template** for launching SEO-focused content/landing domains quickly.  
It combines:
- configurable domain/business metadata (`data/domain.yaml`)
- configurable visual system (`data/theme.yaml`)
- reusable Hugo layouts/partials
- automated article generation (`scripts/generate_article.py`)
- CI/CD publishing workflow (`.github/workflows/publish.yml`)
- analytics + Sedo CTA click tracking

> Current active instance in this repo is configured for **anwaltsagent.de**, but the architecture is intentionally template-first.

## 2) Purpose of each main folder/file

- **`data/`**: Domain-specific runtime configuration that controls behavior and appearance.
- **`layouts/`**: Reusable Hugo rendering logic (templates + partial components).
- **`scripts/`**: Automation tooling for config handling, generation, and validation.
- **`content/posts/`**: Output content generated by pipeline and optionally edited manually.
- **`static/images/`**: Images fetched for generated posts.
- **`.github/workflows/publish.yml`**: Scheduled/manual automation pipeline.
- **`hugo.toml`**: Site identity and Hugo-level config.
- **`docs/DOMAIN_TEMPLATE_REUSE_GUIDE.md`**: Operational guide for spinning up a new domain instance.

---

## 3) How `domain.yaml` works

`data/domain.yaml` is the **domain/business/content source of truth**. The generation script and templates read from it (directly or via helpers).

Typical responsibilities:
- domain identity: `domain`, `brand_name`
- market context: `niche`, `geo`, `language`, `audience`
- positioning and copy blocks used on pages/prompts
- CTA links/labels (including Sedo target URL)
- analytics metadata (`ga_measurement_id`, event names/categories)
- article generation tuning (tone, keyword hints, image hints, CTA text)

How it is consumed:
- `scripts/domain_config.py` loads/normalizes and merges defaults.
- `scripts/generate_article.py` builds prompts and post-processing rules using config values.
- Hugo templates access data via `.Site.Data.domain.*` for runtime rendering.

---

## 4) How `theme.yaml` works

`data/theme.yaml` defines the **design token layer**:
- colors, gradients, border/radius/shadow values
- button/card/section style variables
- typography/label accents

Flow:
1. `layouts/partials/theme-vars.css` maps `theme.yaml` values into CSS custom properties.
2. Page-specific partial styles (`home.css`, `archive.css`, `single.css`) consume those vars.
3. Layout templates include these style partials and stay mostly logic-focused.

Result: you can reskin a new domain with minimal/no template rewrites.

## 5) How `generate_article.py` works

`scripts/generate_article.py` pipeline summary:
1. Load domain config from `DOMAIN_CONFIG_PATH` (default `data/domain.yaml`).
2. Choose keyword (or use `--keyword` override).
3. Build writer/reviewer prompts from domain context (brand/niche/geo/language/tone/SEO hints).
4. Generate article via OpenRouter models.
5. Normalize markdown + frontmatter, enforce required fields/cleanup.
6. Ensure branded CTA exists in final body.
7. Query Unsplash with keyword + style hints, score/select best candidate.
8. Save post markdown to `content/posts/` and image to `static/images/`.

Environment variables:
- `OPENROUTER_KEY` (required for live generation)
- `UNSPLASH_KEY` (required for live image fetching)

## 6) How to use `--dry-run`

`--dry-run` lets you validate config/prompt assembly without calling external APIs or writing output files.

Examples:

```bash
python scripts/generate_article.py --dry-run
python scripts/generate_article.py --dry-run --keyword "anwalt finden"
DOMAIN_CONFIG_PATH=data/examples/domain.legalmatch360.de.yaml python scripts/generate_article.py --dry-run
```

Dry-run is useful for:
- verifying that a new domain preset loads correctly
- reviewing prompt context before production generation
- debugging config changes locally/CI safely

## 7) How `validate_config.py` works

`scripts/validate_config.py` performs structural checks for required keys in:
- `data/domain.yaml`
- `data/theme.yaml`

Behavior:
- exits non-zero when required keys are missing
- prints a success message when both files are valid
- designed for local preflight + CI enforcement in workflow

Run locally:

```bash
python scripts/validate_config.py
```

## 8) How the GitHub Actions workflow works

Workflow file: `.github/workflows/publish.yml`

High-level behavior:
- trigger: schedule + manual `workflow_dispatch`
- optional dispatch input: `domain_config_path`
- setup Python dependencies
- validate config (`scripts/validate_config.py`)
- run article generation (using `DOMAIN_CONFIG_PATH`)
- stage content/image changes and commit only when there are changes
- run Hugo smoke build (`hugo --gc --minify`) for template integrity
- extract domain metadata (via helper script) for notifications
- send status email using configured Gmail secrets

This makes content production repeatable while allowing per-domain config overrides.

## 9) How analytics and Sedo click tracking are wired

- Global analytics partial: `layouts/partials/analytics.html`
- Included globally through shared head partial: `layouts/partials/head.html`
- GA measurement ID comes from `data/domain.yaml`.

Sedo click tracking flow:
1. Sedo CTA links are marked with `data-track="sedo-cta"`.
2. Delegated click handler in analytics partial listens for those clicks.
3. Handler emits configured GA event (`event_name`, `event_category`, domain label) from `domain.yaml` analytics settings.

This gives portable conversion tracking across all templates/pages.

## 10) How to launch a new domain from this template

1. Create a new branch/repo instance from this template.
2. Copy/edit domain config:
   - `data/domain.yaml` (or start from `data/examples/domain.*.yaml`)
3. Copy/edit theme config:
   - `data/theme.yaml` (or start from `data/examples/theme.*.yaml`)
4. Update `hugo.toml` (`baseURL`, `title`, optionally language metadata).
5. Validate configs:
   - `python scripts/validate_config.py`
6. Run generator dry-run:
   - `python scripts/generate_article.py --dry-run`
7. Configure repository secrets (`OPENROUTER_KEY`, `UNSPLASH_KEY`, mail creds).
8. Trigger publish workflow manually for first post.
9. Verify deployed pages, CTA links, GA events, and generated content quality.

---

## 11) Reusable core logic vs domain-specific config

### Reusable core logic (keep/shared)
- Hugo template architecture (`layouts/index.html`, `layouts/posts/list.html`, `layouts/_default/single.html`)
- reusable partial components in `layouts/partials/`
- theme variable bridge (`layouts/partials/theme-vars.css` + style partials)
- config loader/validator scripts (`scripts/domain_config.py`, `scripts/validate_config.py`)
- generation engine (`scripts/generate_article.py`) and workflow orchestration (`.github/workflows/publish.yml`)

### Domain-specific configuration (replace per launch)
- `data/domain.yaml`
- `data/theme.yaml`
- most brand text/CTAs/positioning
- analytics IDs and event naming
- `hugo.toml` site identity (`baseURL`, `title`)
- content produced into `content/posts/` and associated images

## Quick local commands

```bash
# Validate config files
python scripts/validate_config.py

# Preview generation inputs (no API calls)
python scripts/generate_article.py --dry-run --keyword "anwalt finden"

# Live generation (requires API keys)
OPENROUTER_KEY=... UNSPLASH_KEY=... python scripts/generate_article.py

# Hugo local run/build (if Hugo installed)
hugo server -D
hugo --gc --minify
```
